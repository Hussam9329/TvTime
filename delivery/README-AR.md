# TVM-01 + TVM-02 — حزمة توحيد قاعدة البيانات وحالة المحتوى

هذه الحزمة تطبّق التعديلين المطلوبين على مشروع TvTime ضمن مسار واحد متكامل:

- **TVM-01:** اعتماد SQLite + Prisma كمصدر قاعدة البيانات الوحيد، وإلغاء التضارب التشغيلي مع PostgreSQL.
- **TVM-02:** اعتماد `Media.libraryState` كمصدر الحقيقة الوحيد لحالة الفيلم أو المسلسل، وفصل التقييم عن المشاهدة بصورة نهائية.

## القرار المعماري النهائي

المشروع المرفق يعمل على **SQLite فقط** عبر:

```env
DATABASE_URL=file:../db/custom.db
```

السبب أن المشروع الأصلي، سكربتات التشغيل، والقاعدة المرفقة كانت تعمل فعليًا على SQLite، بينما `schema.prisma` كان معلنًا PostgreSQL. الباتش يطابق Prisma مع الواقع التشغيلي بدل إبقاء مسارين متعارضين.

## مصدر الحقيقة الجديد

جدول `Media` هو المصدر الرسمي الوحيد لحالة كل فيلم أو مسلسل. الحقل المعتمد هو:

```text
libraryState
```

القيم المسموحة:

- `none`: العنصر موجود لأسباب مثل التقييم فقط، لكنه غير مضاف للمكتبة وغير مشاهد.
- `planned`: ضمن قائمة المشاهدة لاحقًا.
- `watching`: مسلسل قيد المتابعة.
- `up_to_date`: تمت مشاهدة كل الحلقات المتاحة حاليًا، لكن المسلسل غير منتهٍ.
- `completed`: فيلم مشاهد أو مسلسل مكتمل.

الحقلان القديمان `status` و`watched` بقيا داخل الجدول كمرايا توافقية فقط، ويجري اشتقاقهما مركزيًا من `libraryState`. لا تستخدمهما الصفحات أو الـAPIs لاتخاذ القرار.

## قاعدة التقييم

`userRating` معلومة مستقلة تمامًا عن حالة المشاهدة:

- تقييم فيلم لا يجعله مشاهدًا.
- إزالة التقييم لا تلغي المشاهدة.
- فيلم مشاهد بدون تقييم يبقى `completed`.
- عنصر مقيّم فقط يمكن أن يبقى `none`.

## ترحيل البيانات القديمة

أضيف ترحيل آمن يعمل مرة واحدة ويحوّل البيانات القديمة إلى `Media`:

- `WatchlistItem` ⟵ `planned`
- `FollowingShow` ⟵ `planned` أو حالة متابعة مشتقة من الحلقات
- `WatchedMovie` ⟵ `completed`
- `Rating` ⟵ يملأ `userRating` فقط ولا يغيّر الحالة
- `WatchedEpisode` ⟵ يبقى سجل الحقيقة على مستوى الحلقة، ويرفع المسلسل إلى `watching` أو `up_to_date` عند الحاجة

قبل أي تعديل على الـschema تُنشأ نسخة احتياطية من قاعدة SQLite. كما توجد مرحلة Preflight تدمج أي صفوف `Media` مكررة قبل إنشاء القيد الفريد، حتى لا يفشل الـdeploy على قواعد قديمة.

## أوامر التطبيق الموصى بها

من داخل مجلد المشروع:

```bash
bun install
bun run db:sync
bun run build
```

أو باستخدام npm:

```bash
npm install
npm run db:sync
npm run build
```

الأمر `db:sync` ينفذ بالتسلسل:

1. نسخة احتياطية من SQLite.
2. `prisma generate`.
3. Preflight لدمج التكرارات القديمة.
4. `prisma db push`.
5. ترحيل البيانات القديمة إلى `Media`.
6. فحص سلامة القاعدة والتأكد من عدم وجود حالات متناقضة.

يمكن تشغيل السكربت الجاهز:

```bash
chmod +x delivery/APPLY-COMMANDS.sh
./delivery/APPLY-COMMANDS.sh
```

## تطبيق ملف PATCH فقط

من مجلد نسخة أصلية نظيفة للمشروع:

```bash
patch -p1 < TVM-01-02.patch
chmod +x delivery/APPLY-COMMANDS.sh
./delivery/APPLY-COMMANDS.sh
```

أو استخدم `TVM-01-02.diff` بالطريقة نفسها.

## التراجع عند الحاجة

النسخ الاحتياطية تُخزن داخل:

```text
db/backups/
```

للتراجع:

1. أوقف التطبيق.
2. احتفظ بالقاعدة الحالية باسم آخر.
3. انسخ آخر نسخة احتياطية مكان `db/custom.db`.
4. أعد الكود إلى النسخة السابقة.

لا تشغّل ترحيل النسخة الجديدة فوق قاعدة مسترجعة إذا كنت تريد الرجوع الكامل إلى السلوك القديم.

## ما تغيّر وظيفيًا

- جميع صفحات Library وWatchlist وFollowing وRecently Watched والإحصائيات تقرأ الحالة من `Media.libraryState`.
- جميع عمليات الإضافة والحذف والتقييم والمشاهدة تكتب إلى `Media` من خلال منطق مركزي.
- تعليم موسم كامل صار عملية Bulk واحدة لتجنب سباق الطلبات.
- مشاهدة حلقة لمسلسل غير مضاف تنشئ سجل `Media` تلقائيًا.
- الحلقات المستقبلية لا تُحسب ضمن التقدم ولا يمكن تعليمها كمشاهدة من الواجهة.
- إزالة حلقة من مسلسل مكتمل تعيد حساب حالته فورًا.
- الاستيراد والتصدير وسكربتات الصيانة أصبحت تلتزم بالمصدر المركزي.
- Recently Watched لم يعد يدمج جدولين متضاربين.
- مسار إصلاح المشاهدة العرضية أصبح Dry-run افتراضيًا ولا يغيّر البيانات إلا مع `apply=true`.

## ماذا تفحص بعد التطبيق

راجع بالترتيب:

1. افتح فيلمًا وأضفه إلى Watchlist؛ يجب أن يظهر في كل الصفحات كـ Planned فقط.
2. قيّم فيلمًا غير مشاهد؛ يجب ألا يظهر في Watched أو Recently Watched.
3. علّم فيلمًا كمشاهد بدون تقييم؛ يجب أن يبقى Watched ويظهر في Recently Watched.
4. أزل مشاهدة الفيلم؛ يجب أن يعود إلى Planned مع بقاء تقييمه إن وجد.
5. تابع مسلسلًا وعلّم حلقة واحدة؛ يجب أن يظهر Watching في Library وTV Tracking.
6. علّم كل الحلقات المتاحة لمسلسل مستمر؛ يجب أن يظهر Up to date، وليس Completed.
7. أزل حلقة؛ يجب أن يعود إلى Watching فورًا.
8. قارن نفس العنصر في Home وLibrary وDetails وTracking؛ يجب أن تكون الحالة متطابقة.
9. شغّل `bun run db:verify` أو `npm run db:verify`؛ يجب أن ينتهي بدون أخطاء.
10. أعد تشغيل التطبيق وتأكد أن الحالة لم تعتمد على localStorage ولم تختفِ.

## حدود الفحص داخل بيئة إعداد الحزمة

تم اجتياز:

- فحص Syntax لكل ملفات TypeScript/TSX.
- فحص Syntax لكل سكربتات MJS وShell.
- فحص JSON لـ`package.json`.
- فحص عدم وجود PostgreSQL/Neon في مسار التشغيل.
- فحص عدم وجود قراءة/كتابة Runtime للجداول القديمة، عدا سكربت الترحيل ومسار المسح المقصود.
- محاكاة ترحيل على نسخة معزولة من قاعدة المشروع الأصلية بدون تعديل القاعدة المرفقة.
- اختبار أن Rating-only يبقى `none` ولا يتحول إلى Watched.

لم يكتمل `npm install` داخل بيئة الإعداد بسبب انتهاء مهلة تنزيل الاعتماديات، لذلك لم يتم الادعاء بأن `next build` الكامل نُفّذ هنا. أوامر الحزمة تفرض `db:verify` قبل البناء، ويجب تنفيذها في بيئة المشروع/الـdeploy بعد فك الضغط.
